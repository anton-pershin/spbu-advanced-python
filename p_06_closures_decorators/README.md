# Практика 6: Замыкания и декораторы (`functools`) 

Эта практика посвящена функциям, которые не просто вызываются, а **строят новое
поведение поверх других функций**.

На практике это приводит к двум важным идеям:

- **замыкания** (*closures*) - функции, которые "помнят" значения из внешней
  области видимости;
- **декораторы** - функции, которые принимают одну функцию и возвращают новую,
  добавляя к ней дополнительное поведение.

В качестве небольшой инженерной задачи мы соберем **mini toolkit для
инструментирования функций**: счетчики вызовов, таймер, простое кеширование и
валидацию аргументов.

## Что вы построите

К концу практики у вас будет небольшой пакет `decorators`, который умеет:

- создавать функции с внутренним состоянием через замыкания;
- аккуратно изменять это состояние через `nonlocal`;
- оборачивать функции декораторами;
- сохранять метаданные исходной функции через `functools.wraps`;
- измерять количество и длительность вызовов;
- кешировать результаты для повторяющихся аргументов;
- проверять аргументы до выполнения основной логики.

## Почему это естественное продолжение прошлой практики

В практике 5 функции использовались как значения: их можно было передавать,
сохранять в переменных и частично конфигурировать.

Следующий шаг - научиться **строить новые функции из существующих**.

Именно это делают:

- фабрики функций (`make_counter(...)`, `make_multiplier(...)`);
- декораторы (`@timer`, `@memoize`, `@validate_...`).

То есть идея уже не просто в том, что "функцию можно передать", а в том, что
"функцию можно **сконфигурировать, обернуть и расширить**".

## Предметная область

Представьте, что у вас есть несколько "полезных, но дорогих" функций:

```python
def fib(n: int) -> int:
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)


def parse_score(text: str) -> int:
    return sum(ord(ch) for ch in text.strip())
```

Для таких функций часто хочется добавить дополнительное поведение, не переписывая
их внутреннюю логику:

- измерять время выполнения;
- считать количество вызовов;
- не пересчитывать результат при тех же аргументах;
- заранее отвергать некорректные входные данные.

Если встраивать все это прямо в тело каждой функции, код быстро станет шумным и
трудно переиспользуемым. Декораторы позволяют вынести такую логику в отдельные,
компактные и переиспользуемые компоненты.

## Теория: вложенные функции и lexical scope

Функция в Python может быть объявлена внутри другой функции:

```python
def outer(x: int):
    def inner(y: int) -> int:
        return x + y

    return inner
```

Здесь `inner` использует переменную `x`, хотя `x` не является ее локальным
аргументом. Это возможно потому, что Python хранит ссылку на внешнее окружение,
в котором функция была создана.

```python
add_five = outer(5)
assert add_five(3) == 8
```

Такое поведение и называется **замыканием**.

Практически это означает, что функция может "помнить" параметры, с которыми она
была построена.

## Теория: `nonlocal`

Если внутренняя функция не только читает внешнюю переменную, но и хочет изменить
ее, нужен `nonlocal`.

```python
def make_counter():
    count = 0

    def inc() -> int:
        nonlocal count
        count += 1
        return count

    return inc
```

Без `nonlocal` Python создаст новую локальную переменную `count` внутри `inc`, и
вы получите ошибку или неверное поведение.

Практическое правило:

- если внутренней функции нужно только **читать** значение из внешней области,
  `nonlocal` не нужен;
- если нужно **менять** это значение, используйте `nonlocal` явно.

При этом важно не злоупотреблять скрытым состоянием: замыкания удобны, когда
состояние небольшое и локальное по смыслу.

## Теория: декоратор без синтаксического сахара

Декоратор - это функция, которая принимает функцию и возвращает новую функцию.

```python
def count_calls(func):
    calls = 0

    def wrapper(*args, **kwargs):
        nonlocal calls
        calls += 1
        return func(*args, **kwargs)

    return wrapper
```

Использование без `@` выглядит так:

```python
def square(x: int) -> int:
    return x * x


wrapped = count_calls(square)
wrapped(3)
wrapped(4)
```

То есть мы явно строим новую функцию на основе старой.

## Теория: синтаксис `@decorator`

Запись:

```python
@count_calls
def square(x: int) -> int:
    return x * x
```

эквивалентна:

```python
def square(x: int) -> int:
    return x * x


square = count_calls(square)
```

Важно понимать, что декоратор применяется **в момент определения функции**, а не
в момент ее вызова.

## Теория: зачем нужен `functools.wraps`

Наивный декоратор часто "теряет" информацию об исходной функции:

- `__name__`
- `__doc__`
- `__module__`

Например, без аккуратного оборачивания имя функции может стать просто
`"wrapper"`, что неудобно для отладки, логирования и документации.

Поэтому почти любой практический декоратор должен использовать
`functools.wraps`:

```python
from functools import wraps


def count_calls(func):
    calls = 0

    @wraps(func)
    def wrapper(*args, **kwargs):
        nonlocal calls
        calls += 1
        return func(*args, **kwargs)

    return wrapper
```

## Теория: декораторы с параметрами

Иногда нам нужно настраивать декоратор, например задать имя таймера или правило
валидации. Тогда появляется еще один внешний уровень функции.

```python
def repeat(times: int):
    def decorator(func):
        def wrapper(*args, **kwargs):
            result = None
            for _ in range(times):
                result = func(*args, **kwargs)
            return result

        return wrapper

    return decorator
```

Использование:

```python
@repeat(3)
def greet(name: str) -> str:
    return f"Hello, {name}!"
```

То есть `repeat(3)` сначала строит сам декоратор, а уже потом этот декоратор
оборачивает `greet`.

## Теория: порядок декораторов

Если декораторов несколько, порядок имеет значение.

```python
@timer
@memoize
def fib(n: int) -> int:
    ...
```

Это значит:

```python
fib = timer(memoize(fib))
```

Снаружи находится `timer`, внутри - `memoize`.

Из-за этого:

- один декоратор может видеть уже преобразованное поведение другого;
- логирование, кеширование и валидация могут срабатывать в разном порядке;
- итоговая семантика зависит от композиции, а не только от отдельных частей.

В этой практике это важная часть понимания: декораторы - это не магия, а обычная
композиция функций.

## Задания

### Задание 1. Простые замыкания

В `closures.py` реализуйте несколько фабрик функций.

Минимум стоит поддержать такие идеи:

- функция, которая прибавляет заранее зафиксированное число;
- функция-счетчик, которая хранит число вызовов;
- функция-накопитель, которая помнит предыдущие обновления состояния.

Цель задания - почувствовать, что состояние может жить не только в объекте
класса, но и в замыкании.

### Задание 2. Состояние через `nonlocal`

Реализуйте замыкание, в котором внутренняя функция **меняет** переменную из
внешней области видимости.

Подходящие примеры:

- счетчик вызовов;
- накопленная сумма;
- бегущее среднее.

Важно корректно использовать `nonlocal` и не прятать состояние в глобальных
переменных.

### Задание 3. Базовый декоратор

В `wrappers.py` реализуйте простой декоратор, который добавляет поведение до или
после вызова исходной функции.

Например:

- считает число вызовов;
- печатает краткий trace;
- сохраняет информацию о последнем результате.

Главная цель - увидеть, что декоратор это обычная функция `func -> wrapper`.

### Задание 4. `functools.wraps`

Сделайте так, чтобы ваши декораторы не ломали метаданные исходной функции.

Проверяйте как минимум:

- `decorated.__name__`
- `decorated.__doc__`

Практическое требование: почти все декораторы в этой работе должны быть
реализованы через `@wraps(func)`.

### Задание 5. Декоратор-таймер

Реализуйте декоратор `timer`, который измеряет длительность вызова функции.

Договоритесь, что именно он делает:

- печатает длительность;
- или сохраняет статистику в атрибуте обертки;
- или возвращает результат как обычно, но дополнительно позволяет получить метрику.

Главное - сохранить понятный API и не менять контракт исходной функции.

### Задание 6. Простое кеширование

В `cache.py` реализуйте декоратор `memoize` для функций с hashable-аргументами.

Идея:

- если функция уже вызывалась с теми же `args/kwargs`, возвращаем готовый
  результат;
- если нет - вычисляем, сохраняем в кеш и возвращаем.

Это особенно полезно на примере рекурсивных функций вроде Fibonacci.

При проектировании продумайте:

- как строится ключ кеша;
- какие ограничения у такого подхода;
- что произойдет с mutable или unhashable аргументами.

### Задание 7. Валидация аргументов

В `validation.py` реализуйте один или несколько декораторов для проверки входных
аргументов.

Примеры:

- аргумент `n` должен быть положительным целым числом;
- строковый аргумент не должен быть пустым;
- число должно лежать в допустимом диапазоне.

Идея здесь такая: проверки на границе вызова часто удобнее вынести в отдельный,
переиспользуемый слой, чем дублировать в теле каждой функции.

### Задание 8. Композиция декораторов

Примените несколько декораторов к одной и той же функции и разберите, как влияет
порядок.

Например, сравните:

```python
@timer
@memoize
def fib(n: int) -> int:
    ...
```

и

```python
@memoize
@timer
def fib(n: int) -> int:
    ...
```

Объясните:

- какие вызовы реально измеряются;
- когда срабатывает кеш;
- в каком порядке выполняются проверки и основная логика.

## Что важно по дизайну

В этой практике особенно важно не просто "чтобы работало", а чтобы API был
понятным.

Обращайте внимание на такие вопросы:

- декоратор только печатает что-то в stdout или еще и хранит статистику?
- можно ли использовать его повторно на разных функциях?
- не ломает ли он возвращаемое значение?
- понятно ли по имени декоратора, что именно он делает?

Хороший декоратор обычно:

- сохраняет контракт исходной функции;
- не прячет слишком много магии;
- имеет маленькую и понятную ответственность.

## Частые ошибки

- Забыли `nonlocal`, хотя состояние должно меняться между вызовами.
- Случайно потеряли `__name__` и `__doc__`, потому что не использовали
  `functools.wraps`.
- Храните состояние в глобальной переменной вместо локального замыкания.
- Кешируете вызовы, не подумав о том, что аргументы могут быть не hashable.
- Меняете поведение функции так, что она перестает возвращать ожидаемый результат.
- Не учитываете порядок декораторов при композиции.

## Подсказки

- Для таймера удобно использовать `time.perf_counter()`.
- Для прозрачного оборачивания почти всегда нужны `*args, **kwargs`.
- Если хочется сохранить статистику, можно хранить ее в атрибутах функции-обертки.
- Для первого варианта `memoize` лучше ограничиться только hashable-аргументами и
  явно задокументировать это ограничение.

## Идея для самопроверки

После реализации полезно проверить себя на таких вопросах:

- чем замыкание отличается от обычной функции без состояния?
- зачем нужен `nonlocal` и когда без него можно обойтись?
- что именно делает `@decorator` на уровне присваивания?
- почему `wraps` почти всегда обязателен в практических декораторах?
