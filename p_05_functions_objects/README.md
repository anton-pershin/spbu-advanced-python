# Практика 5: Функции как объекты и функциональные приемы (dependency planner)

Эта практика посвящена функциям в Python как *значениям*, которые можно:

- передавать в другие функции;
- сохранять в переменных;
- частично конфигурировать;
- использовать для настройки алгоритмов без переписывания самих алгоритмов.

В качестве небольшой алгоритмической задачи мы реализуем **dependency planner** -
планировщик задач с зависимостями.

## Что вы построите

К концу практики у вас будет небольшой модуль `planner`, который умеет:

- принимать список задач с зависимостями;
- строить внутреннее представление ориентированного ациклического графа;
- проверять наличие циклов;
- строить topological order;
- разбивать задачи на "слои исполнения" (что можно запускать параллельно);
- настраивать выбор очередной готовой задачи через функцию-стратегию;
- фильтровать и агрегировать задачи через `Callable`, `any/all`, `partial`, `reduce`.

## Почему здесь уместны функции как объекты

У dependency planner есть несколько мест, где поведение удобно задавать **функцией**:

- как выбирать следующую готовую задачу (`ready_key`);
- какие задачи отфильтровать для отчета (`predicate`);
- как агрегировать статистику (`reduce`);
- как преднастроить часто используемые проверки (`functools.partial`).

Идея практики: алгоритмы и структуры данных остаются центральными, а функции
используются как способ **параметризовать поведение**.

## Предпосылки

Ожидается, что вы уже знаете:

- классы и dataclass'ы;
- словари, множества, списки, `deque`;
- что такое граф зависимостей;
- topological sort на базовом уровне.

## Предметная область

Есть задачи вида:

```python
tasks = [
    {"id": "parse", "duration": 2, "deps": []},
    {"id": "build_index", "duration": 4, "deps": ["parse"]},
    {"id": "run_queries", "duration": 3, "deps": ["build_index"]},
    {"id": "report", "duration": 1, "deps": ["run_queries"]},
]
```

Это означает:

- `parse` можно запускать сразу;
- `build_index` зависит от `parse`;
- `run_queries` зависит от `build_index`;
- `report` зависит от `run_queries`.

Если в зависимостях есть цикл, корректного порядка выполнения не существует.

## Теория: функции как значения

В Python функция - это обычный объект.

```python
def square(x: int) -> int:
    return x * x


f = square
assert f(5) == 25
```

Это позволяет писать функции, которые принимают другие функции:

```python
def apply_to_all(xs: list[int], func):
    return [func(x) for x in xs]
```

В этой практике такими передаваемыми функциями будут:

- функция сортировки готовых задач;
- предикаты фильтрации;
- функции агрегации.

## Теория: `Callable`

Если мы хотим явно сказать type checker'у, что ожидаем функцию, используем
`Callable`.

Например:

```python
from typing import Callable


TaskKey = Callable[[Task], object]
Predicate = Callable[[Task], bool]
```

Это делает интерфейс функции более явным: мы не просто принимаем "что-то", а
ожидаем объект, который можно вызвать с `Task`.

## Теория: `lambda`

`lambda` - это способ быстро создать маленькую анонимную функцию прямо в том
месте, где она нужна.

```python
by_duration = lambda task: task.duration
assert by_duration(task) == task.duration
```

Чаще всего `lambda` используют, когда функция:

- очень короткая;
- нужна ровно в одном месте;
- хорошо читается прямо рядом с вызовом `sorted`, `map`, `filter` или другой
  функции высшего порядка.

Например:

```python
sorted(tasks, key=lambda task: (task.duration, task.task_id))
```

Здесь `lambda` удобна, потому что логика ключа маленькая и локальная: сначала
сортируем по длительности, а при равенстве - по идентификатору.

Важно помнить, что тело `lambda` в Python - это **одно выражение**, а не набор
инструкций. Поэтому для коротких преобразований `lambda` удобна, а для более
сложной логики обычно лучше написать обычную функцию через `def`:

```python
def task_sort_key(task: Task) -> tuple[int, str]:
    return task.duration, task.task_id
```

Практическое правило такое:

- `lambda` хороша для коротких локальных функций-ключей и предикатов;
- если логика переиспользуется, становится длинной или требует комментария,
  лучше `def`;
- цель не в том, чтобы везде писать `lambda`, а в том, чтобы код оставался
  читаемым.

## Теория: `sorted(key=...)`

Очень часто алгоритм уже реализован, но хочется менять **политику выбора**.

Например, если несколько задач готовы одновременно, можно выбирать:

- по имени;
- по длительности;
- по числу исходящих ребер;
- по любой другой функции-ключу.

Именно это делает аргумент `key` в `sorted(...)`.

```python
sorted(tasks, key=lambda task: (task.duration, task.task_id))
```

## Теория: `map`, `filter`, comprehension

`map` и `filter` особенно полезны не сами по себе, а тогда, когда у вас уже есть
**готовая функция**, которую хочется передать в конвейер без дополнительной
обвязки.

Например, если преобразование уже оформлено как отдельная функция:

```python
parsed_tasks = list(map(parse_task, payloads))
```

Здесь `map(parse_task, payloads)` хорошо передает идею: "применить один и тот же
разбор ко всем входным payload'ам".

Аналогично, если предикат уже собран заранее, например через `partial`:

```python
from functools import partial

long_task = partial(has_duration_at_least, 5)
long_tasks = list(filter(long_task, tasks))
```

Здесь `filter` подчеркивает, что у нас есть **готовое правило отбора**, которое
мы просто применяем к последовательности.

Еще один хороший случай - когда хочется собрать короткий конвейер из уже готовых
функций:

```python
report_ids = list(
    map(
        lambda task: task.task_id,
        filter(partial(depends_on, "parse"), tasks),
    )
)
```

То есть сначала отфильтровали задачи, зависящие от `parse`, потом спроецировали
их в идентификаторы.

Но во многих случаях comprehension читается лучше:

```python
report_ids = [task.task_id for task in tasks if "parse" in task.deps]
```

Практическое правило такое:

- если уже есть именованная функция или преднастроенный callable, `map/filter`
  могут быть очень уместны;
- если логика маленькая и локальная, comprehension обычно проще читать.

В этой практике важно не "писать только в functional style", а уметь выбирать
подходящий инструмент под конкретную задачу.

## Теория: `any` и `all`

Эти функции особенно удобны для работы с наборами предикатов.

```python
all(predicate(task) for predicate in predicates)
any(predicate(task) for predicate in predicates)
```

Примеры:

- задача удовлетворяет **всем** ограничениям;
- задача удовлетворяет **хотя бы одному** правилу;
- все зависимости уже выполнены.

## Теория: `functools.partial`

`partial` позволяет заранее зафиксировать часть аргументов функции и получить
новую, более специализированную функцию.

### Пример 1. Фиксируем один позиционный аргумент

```python
from functools import partial


def has_duration_at_least(min_duration: int, task: Task) -> bool:
    return task.duration >= min_duration


long_task = partial(has_duration_at_least, 5)
```

Теперь `long_task(task)` - это уже готовый предикат.

### Пример 2. Фиксируем один именованный аргумент

`partial` умеет фиксировать и keyword arguments:

```python
from functools import partial


def topological_order(
    tasks: list[Task], *, ready_key=None, reverse: bool = False
) -> list[Task]:
    ...


reverse_topological = partial(topological_order, reverse=True)
```

Теперь `reverse_topological(tasks, ready_key=...)` всегда вызывает
`topological_order(..., reverse=True)`.

Это особенно удобно, когда вы хотите зафиксировать не "основной" аргумент, а
какой-то режим работы функции.

### Пример 3. Фиксируем позиционный и именованный аргументы одновременно

```python
from functools import partial


def select_tasks(
    tasks: list[Task], predicates: list[Predicate], *, mode: str = "all"
) -> list[Task]:
    ...


select_long_tasks = partial(
    select_tasks,
    predicates=[partial(has_duration_at_least, 5)],
    mode="all",
)
```

Теперь `select_long_tasks(tasks)` - это уже готовая функция, которая применяет
заранее выбранный набор предикатов и заранее выбранный режим.

### Пример 4. Нужно зафиксировать не первый позиционный аргумент

Здесь есть важный практический момент: `partial` подставляет позиционные
аргументы **слева направо**. Поэтому так:

```python
def priority(task: Task, weight: int, bias: int) -> int:
    return weight * task.duration + bias


f = partial(priority, 10)
```

означает не "зафиксировать `weight=10`", а "подставить `10` на место `task`".

Если нужный аргумент можно передать по имени, лучше сделать именно так:

```python
weighted_priority = partial(priority, weight=10)
result = weighted_priority(task, bias=3)
```

Если же функция принимает аргументы только позиционно или нужное место неудобно
фиксировать через keyword, обычно делают маленькую обертку:

```python
def bind_second(func, value):
    return lambda first, third: func(first, value, third)


weighted_priority = bind_second(priority, 10)
result = weighted_priority(task, 3)
```

То есть практическое правило такое:

- если можно, фиксируйте не первый аргумент через keyword;
- если нельзя, проще всего написать маленькую явную функцию-обертку;
- не стоит пытаться насильно решать все случаи одним только `partial`.

Это удобно, когда хочется не писать много похожих маленьких функций вручную, но
при этом важно помнить, как именно `partial` подставляет аргументы.

## Теория: `functools.reduce`

`reduce` последовательно "сворачивает" последовательность в одно значение.

```python
from functools import reduce


total = reduce(lambda acc, task: acc + task.duration, tasks, 0)
```

Но интереснее становится, когда аккумулятор - не просто число, а небольшая
структура данных.

Например, можно одной сверткой посчитать, сколько задач имеет каждую длительность:

```python
from functools import reduce


def add_duration_count(acc: dict[int, int], task: Task) -> dict[int, int]:
    acc[task.duration] = acc.get(task.duration, 0) + 1
    return acc


counts = reduce(add_duration_count, tasks, {})
```

Или собрать статистику по тому, сколько зависимостей имеют задачи:

```python
def add_dep_stats(acc: dict[str, int], task: Task) -> dict[str, int]:
    if len(task.deps) == 0:
        acc["root"] += 1
    else:
        acc["non_root"] += 1
    return acc


stats = reduce(add_dep_stats, tasks, {"root": 0, "non_root": 0})
```

В обоих случаях идея одна и та же:

- есть аккумулятор `acc`;
- каждая новая задача обновляет его;
- в конце получается один агрегированный результат.

В этой практике `reduce` можно естественно использовать для:

- суммарной длительности;
- подсчета количества задач по группам;
- построения небольших агрегированных отчетов.

Важно: `reduce` полезен тогда, когда операция свертки действительно выглядит как
единое повторяющееся правило обновления аккумулятора. Если логика разрастается,
обычный цикл часто читается лучше.

## Теория: topological sort

Граф зависимостей задач - это ориентированный граф:

- вершина - задача;
- ребро `u -> v` означает, что `v` зависит от `u`.

Если граф ациклический, можно построить **топологический порядок** - такой
порядок вершин, в котором каждая задача идет после всех своих зависимостей.

Классический способ - алгоритм Кана:

1. посчитать `indegree` для каждой вершины;
2. положить все вершины с `indegree == 0` в очередь готовых задач;
3. извлекать готовую вершину, добавлять ее в ответ;
4. уменьшать `indegree` у ее потомков;
5. если у потомка `indegree` стал равен нулю - он готов.

Если в конце обработаны не все вершины, значит в графе есть цикл.

специализированные функции.

## Частые ошибки

- путать направление ребра в графе зависимостей;
- не проверять, что зависимость ссылается на существующую задачу;
- делать недетерминированный порядок среди нескольких готовых задач;
- забывать, что `partial` фиксирует аргументы слева направо;
- использовать `reduce` там, где простой цикл был бы существенно понятнее;
- смешивать алгоритмическую часть (topological sort) и форматирование отчета.
