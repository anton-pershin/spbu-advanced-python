# Практика 4: Модели данных и валидация (мини-поисковик)

Эта практика посвящена моделированию данных в Python на примере небольшой
поисковой системы по текстовым документам.

С точки зрения алгоритмов здесь есть вполне классическая задача:

- есть набор документов;
- нужно построить **обратный индекс** (*inverted index*);
- нужно по текстовому запросу вернуть top-`k` наиболее релевантных документов.

С инженерной точки зрения задача тоже реалистична:

- данные приходят извне в виде словарей;
- эти словари нужно валидировать;
- после валидации их удобно преобразовать во внутренние объекты;
- поверх этих объектов строится алгоритм поиска.

Поэтому эта практика естественно объединяет:

- `TypedDict` - описание формы входных словарей для type checker'ов;
- Pydantic - runtime-валидацию и парсинг входных данных;
- `@dataclass` - компактное описание внутренних моделей;
- `property` - контролируемое изменение состояния и проверку инвариантов.

## Что вы построите

К концу практики у вас будет небольшой `SearchEngine`, который умеет:

- принимать сырые payload'ы документов;
- валидировать их через Pydantic;
- превращать их во внутренние модели;
- строить обратный индекс `term -> postings`;
- принимать поисковый запрос и возвращать top-`k` результатов.

## Почему здесь нужны разные виды моделей

В реальных проектах редко бывает одна универсальная структура данных на все
случаи жизни.

Обычно есть как минимум два уровня:

1. **внешние данные** - то, что пришло из файла, HTTP-запроса, очереди или API;
2. **внутренние модели** - то, с чем удобно работать алгоритмам и бизнес-логике.

Причины разделять их:

- внешние данные могут быть неполными или некорректными;
- внешние данные часто естественно выглядят как словари;
- внутренним алгоритмам удобнее работать с явными объектами и инвариантами;
- валидацию лучше делать на границе системы, а не размазывать по всему коду.

В этой практике поток данных такой:

```text
raw dict payload
    -> TypedDict (описание ожидаемой формы)
    -> Pydantic model (runtime validation)
    -> dataclass domain model
    -> inverted index + retrieval algorithm
```

## Предметная область

Мы строим очень упрощенный поисковый движок.

Есть документы:

```python
{"id": "d1", "text": "graph shortest path"}
{"id": "d2", "text": "cache eviction policy"}
{"id": "d3", "text": "graph graph path"}
```

Есть запрос:

```python
{"text": "graph path", "top_k": 2}
```

Система должна:

- провалидировать входные данные;
- разбить тексты на токены;
- построить индекс;
- посчитать score для документов;
- вернуть лучшие результаты.

## Теория: `TypedDict`

`TypedDict` нужен, когда данные по смыслу являются словарем с фиксированным
набором ключей.

Пример:

```python
from typing import TypedDict


class DocumentPayload(TypedDict):
    id: str
    text: str
```

Что это дает:

- `mypy` понимает, какие ключи ожидаются;
- IDE лучше подсказывает структуру данных;
- код становится самодокументируемым.

Что это **не** дает:

- `TypedDict` сам по себе не проверяет runtime-значения;
- словарь `{"id": 123, "text": None}` не станет автоматически ошибкой только
  из-за наличия `TypedDict`.

Поэтому `TypedDict` - это в первую очередь инструмент **статической типизации**.

## Теория: Pydantic

Pydantic решает другую задачу: **runtime validation**.

Пример:

```python
from pydantic import BaseModel


class DocumentIn(BaseModel):
    id: str
    text: str
```

Теперь можно сделать:

```python
doc = DocumentIn.model_validate({"id": "d1", "text": "hello world"})
```

Если данные подходят под схему, вы получите объект модели. Если нет - Pydantic
поднимет ошибку валидации.

В этой практике Pydantic нужен на границе системы:

- проверить, что обязательные поля присутствуют;
- проверить, что строки не пустые;
- проверить, что `top_k > 0`;
- запретить лишние поля.

Практическая идея такая:

- `TypedDict` отвечает на вопрос "какую форму мы ожидаем?";
- Pydantic отвечает на вопрос "корректны ли пришедшие данные на самом деле?".

## Теория: `@dataclass`

`@dataclass` - это удобный способ описывать классы-данные, то есть объекты, у
которых главное - это поля, а не сложное поведение.

Пример без `dataclass`:

```python
class Document:
    def __init__(self, doc_id: str, tokens: tuple[str, ...]) -> None:
        self.doc_id = doc_id
        self.tokens = tokens

    def __repr__(self) -> str:
        return f"Document(doc_id={self.doc_id!r}, tokens={self.tokens!r})"
```

То же самое с `dataclass`:

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class Document:
    doc_id: str
    tokens: tuple[str, ...]
```

`dataclass` автоматически генерирует полезный служебный код:

- `__init__`
- `__repr__`
- сравнение объектов (если нужно)

Почему это полезно именно здесь:

- внутренние модели короткие и декларативные;
- структура объекта видна сразу;
- меньше шаблонного кода;
- инварианты удобно проверять в `__post_init__`.

### `__post_init__`

У `dataclass` есть метод `__post_init__`, который выполняется сразу после
автоматически созданного `__init__`.

Именно там удобно проверять инварианты:

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class SearchQuery:
    tokens: tuple[str, ...]
    top_k: int

    def __post_init__(self) -> None:
        if not self.tokens:
            raise ValueError("query must contain at least one token")
        if self.top_k <= 0:
            raise ValueError("top_k must be positive")
```

В этой практике внутренние dataclass-модели нужны для того, чтобы алгоритм
поиска работал уже не с сырыми словарями, а с хорошо определенными объектами.

## Теория: инварианты

**Инвариант объекта** - это условие, которое должно оставаться истинным для
любого корректного экземпляра.

Примеры инвариантов в этой практике:

- `doc_id` не пустой;
- последовательность токенов документа не пустая;
- `top_k` положителен;
- частота терма в posting'е положительна;
- внутри одного `SearchEngine` идентификаторы документов уникальны.

Если инвариант нарушается, объект либо не должен создаваться, либо должен
немедленно выбрасывать исключение.

## Теория: `property`

Иногда объект в целом должен быть изменяемым, но менять его состояние нужно не
как попало, а под контролем.

Для этого используют `property`.

Пример:

```python
class SearchEngine:
    @property
    def default_top_k(self) -> int:
        return self._default_top_k

    @default_top_k.setter
    def default_top_k(self, value: int) -> None:
        if value <= 0:
            raise ValueError("default_top_k must be positive")
        self._default_top_k = value
```

Зачем это нужно:

- снаружи поле выглядит как обычный атрибут;
- внутри можно проверять корректность значения;
- инвариант поддерживается не только при создании, но и при последующих
  изменениях.

В этой практике `property` используется для настройки `default_top_k` у
`SearchEngine`.

## Теория: токенизация

Поисковый движок не работает с исходными строками напрямую. Сначала текст нужно
разбить на токены.

В этой практике достаточно простой токенизации:

- выделяем последовательности букв и цифр;
- приводим токены к нижнему регистру;
- остальное считаем разделителями.

Подходит регулярное выражение:

```python
r"[0-9A-Za-zА-Яа-яЁё]+"
```

Например:

```python
"Graph, shortest-path!" -> ("graph", "shortest", "path")
```

## Теория: обратный индекс

**Обратный индекс** отображает терм в список документов, в которых он
встречается.

Если документы такие:

- `d1`: `graph path graph`
- `d2`: `cache path`

то индекс можно представить так:

- `graph -> [(d1, 2)]`
- `path -> [(d1, 1), (d2, 1)]`
- `cache -> [(d2, 1)]`

Здесь число - это частота терма в документе.

Почему это хорошая структура данных:

- при поиске по слову не нужно просматривать все документы;
- можно сразу перейти к документам, где слово встречается;
- это один из базовых кирпичиков информационного поиска.

## Теория: ранжирование

Для каждого терма запроса `t` и документа `d` используйте score:

```text
score(d, t) = tf(d, t) * (log((1 + N) / (1 + df(t))) + 1)
```

где:

- `tf(d, t)` - сколько раз терм `t` встречается в документе `d`;
- `df(t)` - в скольких документах встречается терм `t`;
- `N` - общее число документов.

Интуиция:

- если терм часто встречается в конкретном документе, это повышает score;
- если терм встречается почти во всех документах, он менее информативен;
- редкие, но важные термы получают больший вес.

Для multi-term запроса складывайте вклады всех **различных** термов запроса.
Если пользователь написал `graph graph path`, терм `graph` не нужно учитывать
дважды только потому, что он повторился в запросе.

## Частые ошибки

- путать `TypedDict` и runtime-валидацию;
- тащить сырые словари глубоко внутрь алгоритма вместо преобразования их в
  модели;
- забывать, что `dataclass` не отменяет необходимость проверять инварианты;
- не отфильтровывать пустой результат токенизации;
- учитывать повторяющийся терм запроса несколько раз;
- делать недетерминированную сортировку результатов;
- проверять корректность `default_top_k` только в `__init__`, но не в setter'е.
