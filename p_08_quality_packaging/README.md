# Практика 8: Качество кода и упаковка проекта (`pyproject.toml`)

Эта практика посвящена **инженерной стороне Python-разработки**: как превратить
набор скриптов в полноценный проект, который можно установить, протестировать и
проверить на качество.

До этого момента вы:
- запускали код из корня репозитория через `python -m ...`
- использовали `check.py` без `pytest`
- не настраивали инструменты качества

Теперь вы научитесь:
- упаковывать код как устанавливаемый пакет
- запускать тесты через `pytest`
- настраивать `black`, `isort`, `mypy`, `pylint`
- собирать единый quality gate

## Что вы построите

К концу практики у вас будет мини-проект `logtools`, который:
- лежит в структуре `src/`
- устанавливается через `pip install -e .`
- имеет тесты в отдельной директории `tests/`
- конфигурируется через `pyproject.toml`
- проходит единый quality gate

## Почему это важно

Раньше вы работали с кодом, который:
- требовал ручного добавления в `PYTHONPATH`
- не имел четкой границы между "пакетом" и "скриптами"
- проверялся через самописные `check.py`

Теперь вы увидите, как профессиональные проекты:
- изолируют код в `src/` (чтобы тесты не импортировали "мимо")
- используют `pyproject.toml` как единый источник истины
- запускают quality gate перед каждым коммитом

## Как запускать

**Важно**: эта практика устроена иначе, чем предыдущие.

Внутри `template/` и `solution/` находится **отдельный проект** со своей
структурой. Вы работаете из корня репозитория, но команды запускаются внутри
`template/` или `solution/`.

Проверка для шаблона:
```bash
python -m p_08_quality_packaging.template.check
```

Проверка для решения:
```bash
python -m p_08_quality_packaging.solution.check
```

## Структура заготовки

Ваш проект будет находиться в `p_08_quality_packaging/template/`:

```
template/
  pyproject.toml          # конфиг проекта и инструментов
  check.py                # единый quality gate
  src/
    logtools/
      __init__.py         # публичный API
      models.py           # модели данных
      io.py               # чтение/запись логов
      stats.py            # статистика по логам
      service.py          # высокоуровневый API
  tests/
    test_io.py
    test_stats.py
    test_service.py
```

## Предметная область

Вы уже знакомы с обработкой логов из практики 7. Здесь вы соберете похожий, но
более простой пакет `logtools`, который умеет:

- читать лог-файл со строками вида:
  ```text

  2024-05-01T10:01:00,ERROR,disk full
  ```
- парсить строки в объекты `LogRecord`
- считать статистику по уровням

Но теперь фокус не на алгоритмах, а на **качестве упаковки**.

## Теория: `src/` layout

Структура `src/` — это стандартный способ организации Python-проектов:

```
project/
  src/
    yourpackage/
      __init__.py
      module.py
  tests/
  pyproject.toml
```

Преимущества:
- тесты не могут случайно импортировать код "мимо пакета"
- вы тестируете именно установленную версию, а не файлы в директории
- четкая граница между пакетом и вспомогательными файлами

## Теория: `pyproject.toml`

Файл `pyproject.toml` заменяет собой `setup.py`, `requirements.txt`, и конфиги
инструментов.

Минимальная структура:

```toml
[build-system]
requires = ["setuptools>=61", "wheel"]
build-backend = "setuptools.build_meta"

[project]
name = "logtools"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = []

[tool.setuptools.packages.find]
where = ["src"]

[tool.pytest.ini_options]
testpaths = ["tests"]

[tool.black]
line-length = 88

[tool.isort]
profile = "black"

[tool.mypy]
strict = true

[tool.pylint.messages_control]
disable = ["C0114", "C0115", "C0116"]
```

## Теория: editable install

Команда `pip install -e .` устанавливает пакет в режиме разработки:
- код остается в исходной директории
- изменения подхватываются без переустановки
- удобно для разработки и тестирования

## Теория: quality gate

Quality gate — это набор команд, которые должны проходить перед сдачей кода:

- `pytest` — тесты
- `black --check` — форматирование
- `isort --check-only` — импорты
- `mypy` — типы
- `pylint` — стиль и ошибки

Если хотя бы одна команда падает — код не готов.

## Как устроен `check.py`

Файл `check.py` — это обертка над реальными инструментами. Внутри он:
- запускает `pytest`
- запускает `black --check`
- запускает `isort --check-only`
- запускает `mypy`
- запускает `pylint`

Если все проходят — печатает `OK`.

## Частые ошибки

- Запуск `check.py` не из корня репозитория
- Попытка импортировать `logtools` напрямую, а не через `src/`
- Игнорирование ошибок `mypy` или `pylint`
- Отсутствие `__init__.py` в `src/logtools/`
